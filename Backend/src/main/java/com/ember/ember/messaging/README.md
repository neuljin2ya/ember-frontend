# Messaging 패키지 안내

이 패키지는 Spring Boot 백엔드와 FastAPI AI 서버 사이의 비동기 파이프라인을 담당합니다. 발표에서는 "AI 장애가 사용자 기능 전체 장애로 번지지 않도록 어떻게 분리했는가"를 설명하는 핵심 근거입니다.

## 구성

| 하위 폴더 | 역할 | 발표 방어 포인트 |
|---|---|---|
| `config/` | RabbitMQ 큐, 교환기, 라우팅 설정 | 큐를 기능별로 나눠 병목과 장애 범위를 제한합니다. |
| `event/` | 일기 분석, 교환일기 리포트, 라이프스타일, 사용자 벡터 생성 요청 이벤트 | 백엔드 내부 도메인 이벤트를 AI 요청 단위로 명확히 분리합니다. |
| `publisher/` | AI 요청 메시지 발행 | 백엔드 서비스 계층이 RabbitMQ 세부 구현에 직접 묶이지 않게 합니다. |
| `consumer/` | AI 결과 메시지 소비 | 분석 결과를 받아 DB와 후속 도메인 상태에 반영합니다. |
| `outbox/` | Outbox 저장, 재시도, 릴레이 처리 | 트랜잭션 성공 후 메시지가 안정적으로 발행되도록 보장합니다. |
| `idempotency/` | 처리 완료 메시지 기록 | 중복 메시지가 와도 결과가 중복 반영되지 않도록 방어합니다. |

## 질의 대응 예시

- "AI 서버가 죽으면 어떻게 되나요?"
  메인 요청과 AI 추론을 RabbitMQ/Outbox로 분리했습니다. 실패 이벤트는 재시도와 DLQ 관측 대상으로 넘어가며, 사용자의 핵심 플로우는 가능한 범위에서 계속 동작합니다.

- "메시지가 중복 처리되면요?"
  `ProcessedMessage` 기록으로 소비 완료 여부를 관리해 중복 반영 위험을 낮춥니다.

- "왜 그냥 HTTP로 호출하지 않았나요?"
  즉시 응답이 필요한 콘텐츠 스캔은 내부 HTTP를 사용하고, 시간이 걸리는 분석/리포트/벡터 생성은 MQ로 분리했습니다. 기능 특성에 따라 동기/비동기 경계를 나눈 설계입니다.
