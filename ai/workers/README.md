# AI Workers 안내

이 폴더는 RabbitMQ에서 AI 분석 요청을 소비하고 결과를 발행하는 비동기 워커를 담고 있습니다. 발표에서는 "AI 추론을 사용자 요청과 분리해 안정적으로 처리한다"는 점을 설명할 때 사용합니다.

## 주요 파일

| 파일 | 역할 | 발표 방어 포인트 |
|---|---|---|
| `mq_consumer.py` | 일기 분석, 교환일기 리포트, 라이프스타일 분석, 사용자 벡터 생성 큐를 소비 | 큐별 timeout/prefetch를 다르게 두어 무거운 작업과 가벼운 작업을 분리합니다. |
| `mq_publisher.py` | 분석 결과 메시지 발행 | 결과 메시지를 persistent delivery로 발행해 후속 처리 안정성을 높입니다. |

## 큐별 성격

| 큐 성격 | 처리 내용 | 장애 대응 |
|---|---|---|
| 일기 분석 | 단일 일기 감정/성향 분석 | JSON 파싱 실패는 reject, 추론 실패는 DLQ 이동 대상으로 처리합니다. |
| 교환일기 리포트 | 두 사용자의 교환일기 기반 리포트 생성 | 긴 추론 시간을 고려해 timeout/prefetch를 보수적으로 둡니다. |
| 라이프스타일 분석 | 여러 일기의 생활 패턴 집계 | 분석 실패를 메인 백엔드 요청 실패와 분리합니다. |
| 사용자 벡터 생성 | 매칭 추천용 사용자 임베딩 생성 | 추천 품질에 필요한 벡터를 비동기로 갱신합니다. |

## 질의 대응 예시

- "AI 처리량이 늘어나면 어떻게 확장하나요?"
  워커는 큐 기반 소비 구조이므로 인스턴스를 늘려 소비자를 수평 확장할 수 있습니다. 큐별 prefetch도 작업 무게에 맞게 조정합니다.

- "잘못된 메시지가 들어오면 계속 재시도하나요?"
  파싱 실패처럼 재시도해도 복구되지 않는 메시지는 reject로 DLQ 이동시키고, 추론 실패는 nack 기반으로 실패 흐름을 분리합니다.

- "왜 모든 AI 요청을 실시간 HTTP로 하지 않았나요?"
  사용자가 기다려야 하는 검사와 시간이 걸리는 분석을 분리했습니다. 콘텐츠 스캔은 내부 HTTP, 리포트/벡터/라이프스타일 분석은 워커로 처리합니다.
